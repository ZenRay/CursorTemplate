# /tdd-workflow — Test-Driven Development

Complete TDD workflow using Cursor Test Panel and automated testing.

## What is Cursor Test Panel?

Cursor has built-in test integration:
- Shows test results inline in the editor
- Displays pass/fail status in the gutter
- Integrated with Agent for auto-test workflows

## TDD Cycle with Cursor

### 1. Red — Write Failing Test

```
/tdd-workflow red

Describe: "User registration should validate email format"
Test file: src/auth/__tests__/register.test.ts

Agent generates:
- Test case with invalid email
- Assertion for error response
- Edge cases (empty, malformed, missing)
```

Run tests (expect failure):
```bash
npm test -- register.test.ts
```

Cursor Test Panel shows: 🔴 3 tests failing

### 2. Green — Write Minimal Code

```
/tdd-workflow green

Implement minimal validation to pass tests.
Don't over-engineer.
```

Agent generates:
- Email regex validation
- Error response format

Test Panel updates: 🟢 3 tests passing

### 3. Refactor — Improve Code

```
/tdd-workflow refactor

Clean up:
- Extract validation to helper
- Improve error messages
- Add JSDoc
```

Test Panel stays: 🟢 All passing (regression safety)

### Run Affected Tests

After code changes:

```
/tdd-workflow run-affected

Runs only tests related to modified files.
Much faster than full suite.
```

### YOLO Mode Integration

```
/tdd-workflow yolo

Enables:
1. Auto-run affected tests on file save
2. Auto-fix failing tests (if safe)
3. Stop after 3 failed attempts
4. Notify when all pass

⚠️ Requires YOLO mode enabled in settings.
```

## Test Generation Patterns

### Pattern 1: Happy Path + Edge Cases

```typescript
// Generated by /tdd-workflow generate
describe('createUser', () => {
  // Happy path
  it('creates user with valid data', () => {
    const user = createUser({ email: 'test@example.com', name: 'Test' });
    expect(user.id).toBeDefined();
    expect(user.email).toBe('test@example.com');
  });

  // Edge cases
  it('rejects invalid email', () => {
    expect(() => createUser({ email: 'invalid' }))
      .toThrow('Invalid email format');
  });

  it('rejects duplicate email', async () => {
    await createUser({ email: 'test@example.com' });
    await expect(createUser({ email: 'test@example.com' }))
      .rejects.toThrow('Email already exists');
  });

  it('handles empty name', () => {
    const user = createUser({ email: 'test@example.com', name: '' });
    expect(user.name).toBe(''); // or should it throw?
  });
});
```

### Pattern 2: API Endpoint Tests

```typescript
// /tdd-workflow generate-api-test POST /api/users
describe('POST /api/users', () => {
  it('creates user and returns 201', async () => {
    const res = await request(app)
      .post('/api/users')
      .send({ email: 'test@example.com', password: 'secure123' });

    expect(res.status).toBe(201);
    expect(res.body).toHaveProperty('id');
    expect(res.body.password).toBeUndefined(); // never return password
  });

  it('returns 400 for invalid input', async () => {
    const res = await request(app)
      .post('/api/users')
      .send({ email: 'invalid' });

    expect(res.status).toBe(400);
    expect(res.body.errors).toContainEqual(
      expect.objectContaining({ field: 'email' })
    );
  });
});
```

### Pattern 3: Component Tests

```typescript
// /tdd-workflow generate-component LoginForm
describe('LoginForm', () => {
  it('renders email and password inputs', () => {
    render(<LoginForm />);
    expect(screen.getByLabelText(/email/i)).toBeInTheDocument();
    expect(screen.getByLabelText(/password/i)).toBeInTheDocument();
  });

  it('submits form with user input', async () => {
    const onSubmit = jest.fn();
    render(<LoginForm onSubmit={onSubmit} />);

    await userEvent.type(screen.getByLabelText(/email/i), 'test@example.com');
    await userEvent.type(screen.getByLabelText(/password/i), 'password123');
    await userEvent.click(screen.getByRole('button', { name: /login/i }));

    expect(onSubmit).toHaveBeenCalledWith({
      email: 'test@example.com',
      password: 'password123'
    });
  });

  it('shows validation errors', async () => {
    render(<LoginForm />);
    await userEvent.click(screen.getByRole('button', { name: /login/i }));

    expect(screen.getByText(/email is required/i)).toBeInTheDocument();
  });
});
```

```
2. /tdd-workflow green → Implement minimal code

```

```
├── components/
│   ├── Button.tsx
│   └── __tests__/
│       └── Button.test.tsx  # Component tests
├── utils/
│   ├── validation.ts
│   └── __tests__/

├── api/
```

## Best Practices

1. **Test behavior, not implementation** — Refactor-friendly
2. **One assertion per test** (mostly) — Clear failure messages
3. **Descriptive names** — `it('should reject invalid email')`
4. **Given-When-Then** — Structure: arrange, act, assert
5. **Fast feedback** — < 1 second per test file
6. **YOLO with care** — Only in well-tested codebases
